# data/processed/

Model-ready outputs produced deterministically from `data/raw/` and
`data/external/` by the scripts in `data/scripts/`. These files ARE the
published artefact: experiments consume them directly, and any reviewer
should be able to regenerate them byte-for-byte from a clean clone.

## What's committed

| File | Size | Producer | Inputs |
|---|---|---|---|
| `sigbm_canonical.parquet` | 67 KB | `data/scripts/clean_sigbm.py` | `data/raw/sigbm/Relatorio_20260721.csv` |
| `cohort_panel.parquet` | 51 KB | `data/scripts/build_cohort.py` | `sigbm_canonical.parquet` + `data/external/brazilian_failures.csv` |

## `sigbm_canonical.parquet`

The SIGBM registry after canonicalisation by `sentinela.io.sigbm._parse_real_anm`.
Schema (20 columns) is documented at the top of `src/sentinela/io/sigbm.py`.
911 rows, one per registered dam, all categorical fields normalised to
English short-form labels, coordinates in decimal degrees.

## `cohort_panel.parquet`

The (`dam_id`, `month`) panel after cohort filtering and label attachment.
~126,000 rows for the 2014-01 .. 2025-12 window. Columns include the static
engineering features, age-at-month, the binary horizon label `y`, and a
`censored_weight` flag for survival masking. Schema is the contract enforced
by `sentinela.features.build.build_cohort_panel`.

## Regenerating

```bash
python data/scripts/clean_sigbm.py
python data/scripts/build_cohort.py
```

These scripts are deterministic. If the regenerated bytes diverge from what's
committed, that is either an upstream change (new raw file, e.g. fresher
SIGBM export) or a pipeline change — both should be reflected in a
`CHANGELOG.md` entry.
